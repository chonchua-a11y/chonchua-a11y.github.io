Interrogated: is the retention claim real?

The weakest link in the original run was the stickiness claim behind "weekend_warrior isn't churning, they just log less." Re-ran it with PostHog's cohort retention tool instead of the ad-hoc filter used before: power_user holds 100% weekly return across all 13 weeks, weekend_warrior stays in the high 80s–90s% with only a mild dip at the tail. Confirmed — both personas really are retained, this wasn't a counting artifact.

Four things checked

Retention (power_user & weekend_warrior)

confirmed

Re-checked via cohort retention query, not the original stickiness filter. Both personas genuinely show up nearly every week — see above.

Churn / dormancy recovery

unverified

2 churned + 2 dormant out of 186 — but that denominator comes from user_registered, which we now know undercounts. Dormant/churned users don't fire activity_created, so there's no independent way to cross-check this count. Don't cite as settled.

New_user activation funnel

already healthy

95.8% of registered users log a first activity within 30 days, median time to first log is 10h 44m. Real, but scoped to users user_registered actually fired for — see the registration-coverage gap below.

onboarding_completed as an activation signal

dead telemetry

Only 4 events fired in 12 months against 100+ registrations in the last 90 days alone, no custom properties, and no persona value attached on any of the 4. Not wired into the real signup flow — flagged separately, not used here.

New data-quality flag

Chasing the retention discrepancy surfaced a second instrumentation gap: user_registered undercounts the real active population for 6 of 7 personas, sometimes by 2x+ (power_user: 15 registered vs. 34 people actually logging activity). Checked that persona itself isn't the problem — it's stable per user, 0 of 260 active people show more than one value. The gap is real, and it means any earlier analysis sized off user_registered (segment counts, the churn figure above) rests on an incomplete base. Corrected below using activity_created population instead.

What's left: weekend_warrior and casual_runner are both highly retained but log far less per active week than power_user. Using the corrected, activity-based population (260 people, not 186), that's 129 users, about half the base:

Spec

Proposed fix

When a weekend_warrior or casual_runner saves an activity, the product prompts them to log a second one from the same session, so they can capture the training they're already doing without reopening the flow from scratch.

A quick-add flow for logging a second activity in one session

In practice: right after the save confirmation, prompt "log another activity from today?" No new integration, no wearable sync — just remove the friction of opening the log flow twice.

  • Backed by a confirmed, sizeable gap — 129 users, about half the base, using the corrected (activity-based) population count.
  • The retention argument behind this now rests on a proper cohort-retention query, not an ad-hoc one — re-checked and it held up.
  • Shippable in a week. A passive-capture integration (auto-detect from a connected wearable) targets the same problem but is a much bigger lift.
  • Activation checked out fine, no need to fix a funnel that isn't broken. Churn is unverified rather than ruled out — not a reason to hold this, just don't cite it as settled elsewhere.

Reacting to

Open question

The spec assumes weekend_warrior and casual_runner are doing more activity than they log, and that a quick-add nudge would close that gap. Checked Intercom for support messages mentioning missed or forgotten logging — zero results, so this isn't confirmed by real evidence, but it isn't contradicted either, it's genuinely unanswered. Before investing design time in the flow, how would you validate whether these users have more to log, versus this simply being their real activity ceiling?
